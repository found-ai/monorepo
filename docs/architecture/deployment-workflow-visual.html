Deployment promotion workflow

Code integrates on main without deploying a shared environment. Humans explicitly promote a staged, immutable release through two automation-maintained pull requests.

From local change to production

The invariant Staging builds the release candidate once. Production promotes that exact candidate—same manifest, digests, bundles, migration assets, release notes, and affected stack list.

Three rules keep promotion predictable

1

main integrates; it does not deploy

Ordinary PRs get disposable previews. Merging to main updates integration state, versions, and the single staging PR—never a shared environment.

2

Staging creates the release candidate

A human merges the promotion PR. Turborepo selects affected deployables, they build once, and Pulumi deploys the manifest-recorded staging stacks.

3

Production promotes evidence, not source

A human merges the production PR. Automation resolves the successful staging manifest and reuses its exact artifacts without a rebuild or affected recalculation.

What each control contributes

Turborepo · affected selection

The package and task graph selects deployables and transitive dependents. Lockfiles, infrastructure, migrations, schemas, and gateway contracts are modeled inputs. Unknown changes fail closed by selecting all deployables.

Changesets · mandatory intent

Every meaningful PR carries a Changeset. True no-release work uses an explicit empty Changeset or approved exemption. Generated notes and the release plan travel into the immutable manifest.

Pulumi · authoritative GCP state

Pulumi owns shared staging and production state. SST is optional and bounded to local or ephemeral DX; it does not replace environment branches, the manifest, or the lifecycle controller.

Protected branches · human gates

No direct commits land on environment branches. Automation maintains one open PR per hop; a human-approved merge is the only deployment trigger for each shared environment.

Preview reconciler · disposable proof

Internal non-draft PRs receive affected, preview-capable environments. Close, merge, draft, TTL expiry, or failed admission causes idempotent teardown plus absence verification.

Signed manifest · promotion contract

The successful staging record fixes the source SHA, affected deployables, artifact digests, migration assets, release notes, and environment stack lists. Production reads this contract.

Affected query boundaries

# Preview
base = merge-base(PR head, main)
head = tested PR merge SHA

# Staging
base = source SHA in last successful staging manifest
head = candidate source SHA from main

turbo query affected --tasks deploy --base <base> --head <head>

Production deliberately does not query

  • No branch-diff recalculation
  • No cache-based inference
  • No build step
  • Only manifest-recorded artifacts and stacks

The immutable handoff

A successful staging promotion writes a signed manifest keyed to the candidate sourceSha. Artifact references use immutable digests or hashes—not mutable tags.

{
  "sourceSha": "abcdef",
  "affectedDeployables": ["web", "api"],
  "artifacts": {
    "web": { "bundleUri": "gs://…", "sha256": "…" },
    "api": { "imageDigest": "…@sha256:…" }
  },
  "pulumiStacks": {
    "staging": ["web-staging", "api-staging"],
    "production": ["web-production", "api-production"]
  },
  "changesets": ["green-llamas-jump"],
  "status": "succeeded"
}

Promotion evidence

  • Candidate source SHA
  • Affected deployable IDs
  • Images, bundles, and migration assets
  • Staging and production Pulumi stacks
  • Changesets and generated release notes
  • Build run identity and success status

Worked example

PR changes

  • packages/auth
  • services/api
  • One Changeset
  1. CI validates the Changeset. Turborepo selects services/api plus every deployable transitively dependent on packages/auth.
  2. Preview automation creates environments only for affected deployables marked preview-capable.
  3. The PR merges to main. The preview is destroyed and verified absent. No shared environment deploys.
  4. The Changesets version PR updates canonical versions, changelogs, and the release-plan reference on main.
  5. The controller creates or updates the one open main → deploy/staging PR. A human reviews and merges it.
  6. Staging builds affected deployables once, publishes immutable artifacts, writes the manifest, and runs affected Pulumi stacks.
  7. After staging succeeds, the controller creates or updates the one open production PR. A human merges it.
  8. Production deploys the exact staging manifest. It neither rebuilds nor recalculates affected deployables.

Cleanup, failures, and rollback

Preview and staging

  • Preview teardown is idempotent and ends with absence verification.
  • A staging failure does not advance the last successful manifest.
  • Partial deployment fails unless the manifest defines an approved atomicity boundary.

Production and data

  • Retry the same manifest for transient production failures.
  • Rollback redeploys the previous successful application and infrastructure manifest—without rebuilding.
  • Database recovery rolls forward by default. There is no automatic database rollback. Down migrations require explicit, tested support; irreversible changes require approval and a recovery plan.
